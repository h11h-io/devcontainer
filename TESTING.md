# Testing

## Fast local checks

```bash
for test_file in test/unit/test_*.sh; do bash "$test_file"; done
shellcheck --severity=warning src/*/*.sh test/unit/*.sh
shfmt --diff src/*/*.sh test/unit/*.sh
find . -not -path './.git/*' -name '*.json' -print0 |
  xargs -0 -n1 jq empty
```

Unit tests mock downloads, installers, GitHub authentication, TTY behavior, and
shell configuration. They do not require Docker or network access.

## Feature integration

CI uses `@devcontainers/cli` to apply every retained feature and execute its
smoke assertions inside each resulting container. The matrix covers the base
used by our templates and two common language-specific environments:

- `mcr.microsoft.com/devcontainers/base:ubuntu-22.04`
- `mcr.microsoft.com/devcontainers/javascript-node:1-22-bookworm`
- `mcr.microsoft.com/devcontainers/python:1-3.13-bookworm`

Run one feature-image combination locally with:

```bash
devcontainer features test --project-folder . --skip-scenarios \
  -f devbox -i mcr.microsoft.com/devcontainers/base:ubuntu-22.04
```

On Apple Silicon, add `DOCKER_DEFAULT_PLATFORM=linux/amd64` when testing the
Node or Python images; those MCR tags currently publish an AMD64 manifest.

Valid feature IDs are `git-identity-from-github`, `devbox`, and `oh-my-zsh`.
Option scenarios run separately on the standard base image to avoid repeating
equivalent work. Run them with:

```bash
devcontainer features test --project-folder . \
  --skip-autogenerated --skip-duplicated -f <feature>
```

## Base-environment smoke test

### Published features on a local laptop

Run the consumer smoke test to pull the public `:1` feature tags from GHCR,
apply all three features to a clean base image with the `devcontainer` CLI,
and execute assertions inside the resulting container:

```bash
bash test/published-features/run.sh
```

This test requires the `devcontainer` CLI and a running Docker daemon. It uses
`--build-no-cache` so it exercises the published artifacts instead of a cached
feature layer, then removes its test container on exit. It is intentionally a
manual laptop test and does not run in CI.

### Platform smoke test

Build `examples/base/devcontainer.json` from a fixture repository containing a
minimal `devbox.json` with a `setup` script. Verify:

1. `devbox version` works as the remote user without a download;
2. `devbox run setup` resolves the project exactly once;
3. `docker info` reaches the platform daemon;
4. Git identity gracefully configures or reports missing authentication;
5. stopping and restarting the container restores `nix-daemon`.

Run the smoke test in GitHub Codespaces, a Coder/envbuilder workspace with its
Docker socket exposed, and local VS Code. Projects using Docker bind mounts must
also verify that their host and container workspace paths resolve correctly.
